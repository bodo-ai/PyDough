# PyDough Performance Benchmarker

A benchmarking framework for measuring and tracking the performance of the PyDough DSL across standard industry benchmarks (TPC-H, TPC-DS).

## Overview

The Benchmarker collects performance data by executing PyDough DSL code against benchmark queries, validating results against ground truth SQL, and recording execution metrics. This enables regression tracking across releases and surfaces opportunities for new optimizations.

## Automated Runs

The benchmarker runs automatically on every PyDough DSL release via GitHub Actions. Results are committed as a new PR containing a timestamped CSV file. The workflow can also be triggered manually on GitHub or using the `[run benchmark]` flag on a message commit.

The generated `.csv` file has the format described in `Metrics` section. A new file and PR will be created on every run.

---

## Setup

### Requirements

- A questions CSV file (see [Questions File Format](#questions-file-format) below)

### Constructor

```python
from benchmarker import Benchmarker

benchmarker = Benchmarker(
    postgres_conn=conn,           # psycopg2 (or compatible) connection
    questions_path="./",          # directory containing the questions file
    filename="questions",         # questions filename (without .csv extension)
    export_metrics=False,         # write metrics CSV on completion
)
```

| Parameter | Type | Default | Description |
|---|---|---|---|
| `postgres_conn` | `Connection` | — | Active PostgreSQL connection |
| `questions_path` | `str` | `"./"` | Path to the directory holding the questions file |
| `filename` | `str` | `"questions"` | Name of the questions CSV (without extension) |
| `export_metrics` | `bool` | `False` | If `True`, writes the metrics CSV after `measure()` completes |
| `retries` | `int` | `3` | Number of retry attempts per question on transient failures |

---

## Questions File Format

The questions file is a CSV with the following columns:

| Column | Description |
|---|---|
| `benchmark` | Benchmark name (e.g. `TPCH`, `TPC-DS`) |
| `question_id` | Unique number for query identification |
| `pydough_code` | PyDough DSL code that answers the question |
| `postgres_sql` | Ground truth SQL (official benchmark query) |

### Example row (TPC-H Q1)

```
benchmark,question_id,pydough_code,postgres_sql
TPCH,1,<pydough_code>,"SELECT l_returnflag, l_linestatus, SUM(l_quantity) AS sum_qty, ..."
```

## Usage

```python
# Run all benchmarks and (optionally) export the metrics file
benchmarker.measure()
```

`measure()` is the main entry point. It iterates over every row in the questions file, executes both the PyDough code and the ground truth SQL, validates results, records metrics, and — if `export_metrics=True` — writes the output CSV.

---

## Metrics

Each question produces one row in the metrics DataFrame, written to a CSV named `{date}_{R/M}_{pydough_version/run_id}.csv`.
Where `R` means that the benchmark run was trigger by a new release and `M` that was triggered manually.

| Column | Type | Description |
|---|---|---|
| `question_id` | `int` | Unique identifier for the question |
| `benchmark` | `str` | Name of the benchmark suite |
| `ground_truth` | `str` | Reference SQL query |
| `pydough_code` | `str` | Pydough code use to generate the SQL |
| `pydough_sql` | `str` | SQL generated by PyDough |
| `exec_time` | `float` | Execution time of the ground truth SQL (s) |
| `pydough_exec_time` | `float` | Execution time of the PyDough SQL (s) |
| `exec_plan` | `str` | Database execution plan for ground truth query |
| `pydough_exec_plan` | `str` | Database execution plan for PyDough query |
| `status` | `str` | Outcome: `SUCCESS` or `FAILURE - {error message}` |

Execution times are measured using `time.perf_counter()` for high-resolution timing.

---

## API Reference

### `measure() -> None`
Main entry point. Orchestrates the full benchmark run — loads questions, executes both SQL paths, validates results, and collects all metrics.

### `execute_pydough(pydough_code: str) -> tuple[DataFrame, str]`
Executes the given PyDough DSL code using the `from_string` API and returns the result as a DataFrame and the generated SQL.

### `execute_sql(query: str) -> DataFrame`
Executes a raw SQL query against the configured PostgreSQL connection and returns the result as a DataFrame.

### `generate_execute_plan(query: str) -> str`
Runs `EXPLAIN ANALYZE` (or equivalent) on the given query and returns the execution plan as a string.

### `result_validation(pydough_result: DataFrame, ground_truth_result: DataFrame) -> None`
Compares the PyDough result against the ground truth. Supports complex comparison logic to account for ordering and floating-point tolerances.

### `generate_metrics_file() -> None`
Exports the collected metrics to a CSV file named `{date.today()}_{pydough_version}.csv`.
